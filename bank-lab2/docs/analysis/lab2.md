# Лабораторна робота 2

## 1. Що змінилося порівняно з лабораторною 1

У лабораторній роботі 1 використовувалась класична архітектура **MVC (Model-View-Controller)**.  
Проєкт був складнішим з точки зору функціональності, але мав тісно пов’язані компоненти:

- контролери напряму взаємодіяли з сервісами та репозиторіями
- бізнес-логіка частково знаходилась у сервісах або навіть у контролерах
- доменна модель не була чітко відокремлена від інфраструктури

У лабораторній роботі 2 було виконано рефакторинг до шарової архітектури з використанням принципів Domain-Driven Design:

- **Presentation layer** — відповідає лише за HTTP та DTO
- **Application layer** — реалізує use cases
- **Domain layer** — містить бізнес-логіку та інваріанти
- **Infrastructure layer** — реалізує доступ до даних

Таким чином, навіть при меншій функціональній складності, архітектура стала більш правильною з точки зору проєктування.

---

## 2. Переваги шарової архітектури

Розділення на шари дало такі переваги:

- **Ізоляція бізнес-логіки**  
  Domain layer не залежить від Spring, JPA чи HTTP.

- **Краща тестованість**  
  Доменні та application тести можна запускати без БД і без Spring.

- **Гнучкість**  
  Легко змінити реалізацію репозиторію або базу даних, не змінюючи бізнес-логіку.

- **Чітка структура**  
  Код став зрозумілішим: кожен шар має свою відповідальність.

- **Використання DIP (Dependency Inversion Principle)**  
  Domain визначає інтерфейси, а Infrastructure їх реалізує.

---

## 3. Недоліки та ускладнення

Порівняно з лабораторною 1, з’явились наступні складності:

- **Більше коду**  
  Додались окремі класи для DTO, маперів, фабрик, інтерфейсів.

- **Необхідність мапінгу**  
  Потрібно перетворювати:
    - Entity ↔ Domain
    - Domain ↔ DTO

- **Складніша навігація по проєкту**  
  Код розбитий по шарах, що спочатку ускладнює орієнтацію.

---

## 4. Наскільки простіше змінити БД або фреймворк

Завдяки розділенню на шари:

- Domain layer не залежить від JPA → його не потрібно змінювати
- Application layer працює через інтерфейси → також не залежить від конкретної БД
- Змінюється лише Infrastructure layer (репозиторії, entity, мапери)

Це значно спрощує:
- заміну PostgreSQL на іншу базу даних
- зміну способу доступу до даних (наприклад, іншу ORM або JDBC)

- Оскільки доменний шар не залежить від фреймворків, його не потрібно змінювати при таких змінах.

---

## 5. Обраний підхід: Rich Domain Model

Було обрано підхід **Rich Domain Model**, оскільки:

- бізнес-логіка знаходиться всередині доменних моделей (`Account`)
- методи `deposit` та `withdraw` інкапсулюють правила роботи з балансом
- `Money` як Value Object відповідає за коректність операцій з грошима

Це дозволяє:
- уникнути дублювання логіки
- гарантувати виконання інваріантів
- зробити модель більш виразною

---

## 6. Domain Factory

Було реалізовано `AccountFactory`, який відповідає за створення доменних об'єктів.

Фабрика перевіряє прості інваріанти через використання `Money` (наприклад, валідність суми та валюти).

Складні інваріанти (які потребують доступу до БД) не були реалізовані, оскільки в даній предметній області відсутні такі вимоги.

---

## 7. Тестування

Було реалізовано три типи тестів:

### Unit-тести домену
- тестують `Account` та `Money`
- не використовують БД або Spring

### Unit-тести application layer
- тестують `AccountService`
- використовують fake-репозиторій (in-memory реалізацію)

### Integration-тести
- перевіряють REST API
- проходять повний цикл: HTTP → Controller → Service → Repository → DB

## Висновок

У результаті лабораторної роботи було отримано більш гнучку, тестовану та масштабовану архітектуру.  
Хоча структура стала складнішою, вона значно краще відповідає реальним вимогам до backend-систем.
