# Лабораторна робота 4 — Синхронна та асинхронна комунікація між компонентами

## 1. Що змінилося порівняно з лабораторною 3

У лабораторній роботі 3 вся логіка виконувалась послідовно:

```text
Controller → Handler → Domain → Repository
```

Handler виконував операцію і на цьому flow завершувався.

У лабораторній роботі 4 було додано побічні операції, які виконуються після основної бізнес-дії.  
Для цього було реалізовано окремий компонент аудиту та два способи взаємодії з ним:
- синхронний
- асинхронний через події

---

## 2. Побічні операції та допоміжний компонент

У банківській системі після операцій `deposit` і `withdraw` потрібно виконувати аудит дій користувача.

Для цього було створено окремий компонент:

```text
AuditService
```

та його реалізацію:

```text
ConsoleAuditService
```

Компонент відповідає тільки за аудит і не містить бізнес-логіку акаунтів.

---

## 3. Синхронна комунікація

У синхронному варіанті `DepositHandler` після успішного `deposit` напряму викликає:

```java
auditService.log(...)
```

Схема:

```text
DepositHandler -> AuditService
```

Особливості:
- handler знає про `AuditService`
- виклик відбувається в тому ж потоці
- API чекає завершення аудиту

Переваги:
- проста реалізація
- легко дебажити

Недоліки:
- сильніша зв’язаність
- повільніша відповідь API
- помилка audit може вплинути на основну операцію

---

## 4. Асинхронна комунікація

У другому варіанті було реалізовано event-driven підхід.

Після успішного `deposit` handler публікує подію:

```text
MoneyDepositedEvent
```

Подія реалізована як immutable `record` і містить:
- `accountId`
- `amount`
- `currency`

Для доставки подій використовується Spring Event Bus:

```text
ApplicationEventPublisher
```

Подію обробляє:

```text
AuditEventListener
```

Схема:

```text
DepositHandler
    -> publishEvent()
        -> AuditEventListener
```

Listener виконується асинхронно через:

```java
@Async
@EventListener
```

Тому основна операція не чекає завершення аудиту.

---

## 5. Порівняння підходів

| Синхронний | Асинхронний |
|---|---|
| Простий | Складніший |
| Handler залежить від AuditService | Handler не знає listener |
| API відповідає довше | API відповідає швидше |
| Помилка audit впливає на request | Listener може впасти окремо |
| Легше тестувати | Складніше тестувати |

---

## 6. Поведінка при збоях

У синхронному варіанті збій `AuditService` може зламати виконання `deposit`.

В асинхронному варіанті `deposit` вже завершений, тому помилка listener-а не впливає на основну операцію.

---

## 7. Ідемпотентність

У поточній реалізації `AuditEventListener` не є ідемпотентним.

Якщо одна й та сама подія буде доставлена двічі, listener також двічі виконає аудит операції. У моєму випадку це не є критичною проблемою, оскільки аудит лише виводиться в консоль.

У production-системах для вирішення цієї проблеми зазвичай:
- зберігають ID вже оброблених подій
- або використовують механізми deduplication 
Це дозволяє уникнути повторної обробки однієї й тієї самої події.
---

## 8. Тестування

Було реалізовано unit-тест `DepositHandlerTest`.

Тест перевіряє:
- виклик `AuditService`
- публікацію `MoneyDepositedEvent`

Для тестування використовувались:
- fake repository
- Mockito mocks

---

## Висновок

У лабораторній роботі було реалізовано синхронну та асинхронну комунікацію між компонентами.

Синхронний підхід простіший, але створює сильнішу залежність між компонентами.

Асинхронний підхід через Event Bus краще підходить для масштабованих систем, оскільки дозволяє ізолювати побічні ефекти від основної бізнес-логіки.
